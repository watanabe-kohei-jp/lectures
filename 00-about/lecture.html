3 つの解決策 — その 1
01

PPTx ではなく、HTML

AI が直接読めて、編集もできて、Git で履歴が残る

  • AI が構造を理解できる。 PPTx はバイナリ。HTML は構造化テキストなので、Claude Code が そのまま読み・編集できる。
  • Git の差分が取れる。 誰が・いつ・なぜ変えたかが一行単位で追える。バイナリの PPTx では不可能。
  • 生成・編集が圧倒的に速い。 PowerPoint を立ち上げる必要なし。トークン消費も大幅に少なく、コストが安い。

※ HTML は "プログラミング言語" ではないが、機械可読 = AI フレンドリー。Web エコシステム（ブラウザ・Git・CDN）がそのまま使える。

3 つの解決策 — その 2
02

詰め込まず、枚数も増やさない

詳細は読者の AI（Claude Code）に深掘りしてもらう前提

  • スライドは骨格と主張のみ。 詳細を抱え込まない。「ここから先は AI に聞いてください」 と言える形を最初から想定する。
  • 読者が AI に質問する練習になる。 スライドを読み切るのではなく、隣に Claude Code を置いて深掘り。それ自体が学びの一部。
  • 陳腐化に強い。 細かい数値や設定値は変わるが、骨格と主張は長く使える。抽象度の高い情報だけが残る。

※ 1 枚 1 メッセージ・全体で 25 分以内・文字は最小限。枚数を増やさないのが原則。

3 つの解決策 — その 3
03

1 回作って、終わりにしない

Git/GitHub で、誰でも・いつでも更新できる

  • GitHub エコシステムを、そのまま使う。 履歴・差分・Issue・Pull Request・Discussions — 世界中で確立された 共同編集の道具一式 が、自分の資料にそのまま乗る。新しいツールを覚える必要はない。
  • PR を書けないなら、Issue だけでいい。 「ここがわからなかった」「こうしてほしい」を 一行 書くだけで参加できる。プロでなくていい。むしろ、最初に読んだ人の素直な声こそ資料を良くする。
  • セッション JSON も資産。 Claude Code 等のセッションログを保存すれば、「どんな対話で良い修正が生まれたか」 を後から分析できる。

この資料の使い方

古くなったら、あなたが直せる

この資料は完成品ではない。育てるための土台

  • 読み手として： 隣に AI（Claude Code / Codex）を置いて読む。気になるところはその場で深掘り。
  • 意見したくなったら、Issue を立てる。 「ここがわからなかった」「こうしてほしい」「この情報は古い」 — 一行で Issue を立てるだけで参加できる。プロでなくていい。素直な疑問こそ、資料を育てる。
  • 本格的に直したいなら、PR： GitHub から fork して、AI に頼んで HTML を直す。PR で送り返す。マージされたら、あなたの修正がこの資料に乗る。
  • セッションも残せる： Claude Code のセッション JSON を repository に保存すると、後から「どんな対話で良い修正が生まれたか」を学べる。

※ Issue → 読む → 直す → PR → merge — このサイクルで資料は育っていく。詳細は ../CLAUDE.md。
