# Lectures — 持続可能な教育・知識・技術の運用

> 「飢えた人に魚を与えれば一日食べさせられる。<br>
> 魚の釣り方を教えれば一生食べさせられる。」

このリポジトリは "資料を配布する場所" ではなく、**教育・知識・技術を持続可能に運用するための実験場**。Lecture 01〜の各回はその応用例にすぎない。

---

## 基礎思想

### 1. AI 前提で書く・読む・編集する
- これからは、書くのも読むのも編集するのも AI が主役になる
- 人間は「指示する人」、AI が「手を動かす人」
- 資料は人間と AI が **一緒に読み進める** 前提で設計する

### 2. AI フレンドリーな媒体を選ぶ（PPTx より HTML）
- JavaScript → TypeScript の移行と同じ流れ：AI 親和性の高い "型のある" 言語が選ばれていく
- PPTx の本来の目的は **コミュニケーション**。手段は PPTx でなくてもよい
- HTML が PPTx より優れる点：
  - 構造化されており、AI が読みやすい・編集しやすい
  - エコシステム（Git・ブラウザ・Web 標準）がそのまま使える
  - 生成・編集が圧倒的に高速、トークンコストも安い
- だからこのリポジトリのスライドは **HTML ベース**（`shared/deck-stage.js` + `shared/theme.css`）

### 3. 中身は「概要」、詳細は読者の AI へ
- スライドが詳細をすべて抱え込むと、(1) 膨張する、(2) すぐ陳腐化する、(3) 読者が AI に質問する練習機会を奪う
- 資料は骨格と主張だけ。深掘りは読者が手元の AI（Claude Code 等）に頼む前提

## 運用方針

### 1. Git で全履歴を残す
- 誰が・いつ・何を・なぜ変更したか、すべて追える状態にする
- スライドは "凍結された PDF" ではなく "進化する HTML"

### 2. オープンに、誰でも編集できる
- GitHub に置く。読者の AI が直接編集できる前提
- 古い情報・誤りを見つけたら、読者の手で修正・PR してもらう
- "教える側 / 教わる側" の境界をゆるくする

### 3. セッションファイルも保存・共有対象
- Claude Code 等のセッション JSON は、編集の履歴データとして蓄積
- 「どのプロンプトが、どんな良い資料を生んだか」を後から分析可能に
- これは新しい点：従来のドキュメント運用にはない

### 4. 変わるのが前提
- AI 周辺は移り変わりが激しい。古くなったら直す、ではなく、**最初から "更新される前提" で設計**
- HTML × Git × PR ベースの運用は、その変化への対応手段

## このリポジトリで作業するときの指針（Claude 向け）

### コンテンツ
- スライドは **概要のみ** を書く。手順詳細は書かない
- 機能・ツールの話より、**主張・なぜそれが大事か** を優先する
- 文章は短く・断定的に。冗長な接続詞や枕詞を削る
- 章立て・順番は **目的・主張 → 具体理由** の流れを基本にする（歴史・背景は補強として後ろに置く）
- 引用するときは出典（チャンネル名・URL・日付）を併記し、後から読者が辿れるようにする
- 数字を使うときは出所と取得日を明記。古くなる前提で、更新可能な形にする

### スライド構造（必須レイアウトルール）

各スライドは「**今この資料の中でどこにいるか**」が一目でわかる形にする：

1. **左上にチャプター（`.meta-line` の `.accent`）**
   - 例：`<span class="accent">Chapter 3</span> · 立ち位置の変化`
   - 「いま何の話か」を毎スライド明示する

2. **右上にページ番号（`.meta-line` の右）**
   - 例：`05 / 12`
   - 全体での位置感

3. **下部に横長プログレス（`.progress-strip`）**
   - `shared/progress-strip.js` が全スライドに自動注入する
   - 全スライドの短縮ラベルを並べ、現在位置をクレイで強調
   - スライドの **データ参照は `data-label` から自動抽出**（"—" 区切りの前を短縮表示）
   - 各 `<section>` に `data-label="<タイトル> — <サブ>"` を必ず書く

4. **下端に `.brand`（著者・章名）**
   - bottom: 56px の位置。プログレスとは干渉しない

### スクリプト読み込み順
```html
<link rel="stylesheet" href="../shared/theme.css">
<script src="../shared/deck-stage.js"></script>
<script src="../shared/progress-strip.js" defer></script>
```

### CSS / レイアウト
- レイアウトに迷ったら、`theme.css` の既存クラスを優先使用：
  - `.cover`, `.content`, `.meta-line`, `.brand`
  - `.cols-2`, `.cols-3`, `.timeline`, `.matrix`, `.quad`
  - `.stats-grid`, `.roadmap`, `.voices`, `.resources`
  - `.solution-header` (01/02/03 番号付きスライド)
  - `.progress-strip` / `.ps-step` / `.ps-dot` / `.ps-label`
- 新しいレイアウトを追加する場合は `theme.css` に汎用クラスとして登録（インラインスタイルは最小化）

### 確認
- スライドを追加・編集したら、ブラウザで `R`（先頭へ）→ `→` で全枚を通読
- 各スライドの `scrollHeight === 1080` を JS で確認すると overflow しない
- 下部プログレスと brand が干渉していないかチェック

## プロンプトインジェクション対策

このリポジトリの教材は、受講者の AI（Claude Code 等）が読む前提で書く。だからこそ「**AI に伝わる内容は人間にも見える**」を不変条件にする。

### なぜ

受講者の AI は本人の代理人であり、信頼の中心。第三者が PR で hidden な指示を埋め込めば、受講者は知らないうちに他人の指示を自分の AI で実行することになる。

悪意の有無に関係なく禁止。「親切な誰か」が良かれと思って埋めた `data-ai-context` でも、本人の同意なしに AI を動かしている時点で構造的に injection と同じ。

### 不変条件

- AI 向けの命令・補足は **必ず `.ai-buddy`（または同等の可視ブロック）で書く**
- 以下は **PR で禁止**：
  - `data-ai-*` 属性
  - `<template>` タグ
  - `display:none` / `visibility:hidden` で隠した文字
  - HTML コメント（`<!-- ... -->`）の中の AI 向けキーワード
  - JSON-LD `<script type="application/ld+json">`
  - Zero-width Unicode（U+200B 等）を使った隠し文字

### 強制

`.github/workflows/prompt-injection-check.yml` が PR の追加行を上記パターンで自動スキャンし、検出したら CI を fail させる。

- スキャン対象は **追加行のみ**（既存コードは grandfather）
- 例外が必要な場合は Discussion で先に議論する
