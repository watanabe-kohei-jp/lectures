Lecture 03 つかみ
Chapter 1 · LLM は何をしているか 03 / 10
機能の解説 — その 1

LLM は 次の token を予測している、それだけ

  • token は単語より小さい単位。「Claude」「,」「 a」など、語の断片。日本語は約 1〜2 文字で 1 token。
  • LLM は「これまでの token 列の次に来る token」を確率で予測する。1 個出力したら、それを含めて次を予測。これを繰り返して文章になる。
  • "見える範囲" は有限。一度に扱える token 数 = コンテキストウィンドウ。Claude Sonnet 4.6 / Opus 4.7 で約 200,000 token。
  • ウィンドウに入っていないものは "存在しない"。LLM にとっては、見えないファイル・忘れた会話は文字通り無いも同然。

※ 「賢く考えている」ように見えるが、内側はひたすら次の token を確率で並べているだけ。だから「何を見せるか」が出力を決める。

Chapter 2 · プロンプトの 3 役 04 / 10
機能の解説 — その 2

LLM が受け取る入力 = プロンプト には 3 つの役がある

役（role）
役割
中身の例
system
ふるまいの指示（前提・方針）
「あなたは Claude Code。安全に・正直に・簡潔に答える」
user
ユーザーからの入力
「この関数のバグを直して」
assistant
過去の LLM の応答
「以下のように修正しました…」

※ Claude Code はこれらを自動で構築して LLM に送る。CLAUDE.md は system role に、会話履歴は user/assistant role に乗る。

Lecture 03 Chapter 2
Chapter 3 · コンテキストエンジニアリング 05 / 10
機能の解説 — その 3

必要な情報を、必要なときに 見せる技術

プロンプトエンジニアリングの後継概念。Claude Code を含む agent 時代の中心スキル

  • 「全部詰め込む」が答えではない。長くなるほど attention が希薄になり、肝心な情報が埋もれる。
  • 古い情報は害になる。セッションが長引くと、関係ない過去の会話がノイズになって判断を歪める。
  • "見せる" と "見せない" の選択がコア。何を常駐させ、何を都度渡し、何を別エージェントに任せるか。

※ Claude Code の機能（CLAUDE.md・skills・slash commands・subagent）は、すべて コンテキストエンジニアリングの実装。次の応用で見る。

Lecture 03 Chapter 3
応用 · CLAUDE.md 06 / 10
使いどころ — その 1

CLAUDE.md = 常駐 させる情報の置き場

毎セッションで自動的にコンテキストに乗る = 説明コストがゼロになる

コンテキストエンジニアリング的に言うと

「このプロジェクトを触る限り、常に見えていてほしい」情報をsystem role に固定する仕組み。

技術スタック / 禁止事項 / よく使うコマンド / プロジェクトのフェーズなどが該当。

注意

常駐するということは、コンテキストウィンドウの一部を常に消費するということ。短く・本当に必要なものだけ書く。

シークレット（API キー等）は絶対に書かない。

※ 書き方・置き場所・効果的なパターンは 第 4 章：CLAUDE.md でプロジェクトを記憶させる で詳しく。

Lecture 03 応用 · CLAUDE.md
応用 · skills と subagent 07 / 10
使いどころ — その 2

指示を パッケージ化 し、重い作業は 分離 する

skills / slash commands

定型の指示をパッケージ化する仕組み。「コードレビューして」「セキュリティ観点で点検」など、繰り返し使うプロンプトを名前で呼べる。

コンテキスト的には「必要なときだけ展開される指示」。常駐させずに済む。

subagent

重い調査や独立した作業を 別のコンテキスト で動かす仕組み。例：コードベース全体の探索、別観点でのレビュー。

コンテキスト的には「メインの会話を汚さずに重い作業を任せる」。終わったら要点だけ戻ってくる。

※ どちらも「本当に必要なものだけメインのコンテキストに乗せる」というコンテキストエンジニアリングの原理を実装した道具。
